✅ NPM Overview for DA1

Node Package Manager (NPM) is a critical tool in the DA1 development ecosystem. We use NPM to:

  • Manage all JavaScript dependencies
  • Run standardized build, test, and development scripts
  • Ensure consistent development environments across the team
  • Automate repetitive development tasks

All DA1 JavaScript projects use NPM with consistent standards to maintain code quality and project organization.

✅ Installation and Setup

Required Versions

For all DA1 projects, we standardize on:

  • Node.js version 18.x or higher
  • NPM version 8.x or higher

Installation Verification

After installing Node.js and NPM, verify your installation with:

node --version
npm --version

Project Initialization

When creating a new DA1 project:

  1. Create a new directory for your project
  2. Initialize NPM with npm init
  3. Provide accurate project information during initialization
  4. Set the license to "private" unless specifically instructed otherwise

✅ Dependency Management

Adding Dependencies

When adding dependencies, always specify the exact version:

# Production dependency
npm install express@4.17.1 --save

# Development dependency
npm install jest@27.0.6 --save-dev

Dependency Types

Type Flag Purpose Example
Production --save or no flag Required for runtime express, axios, mongoose
Development --save-dev Only needed for development jest, eslint, nodemon
Global -g Tools available system-wide npm, nodemon, pm2

Dependency Auditing

Regularly audit your dependencies for security vulnerabilities:

npm audit

Fix security issues with:

npm audit fix

✅ NPM Scripts Standards

All DA1 projects should include the following standard scripts in their package.json:

"scripts": {
  "start": "node index.js",
  "dev": "nodemon index.js",
  "test": "jest",
  "lint": "eslint .",
  "build": "webpack --mode production"
}

Script Purposes

  • start: Runs the production server
  • dev: Runs development server with auto-restart
  • test: Runs all unit and integration tests
  • lint: Checks code quality against our standards
  • build: Creates production-ready assets

Custom Scripts

Project-specific scripts should use descriptive names that clearly indicate their purpose:

"scripts": {
  // ... standard scripts ...
  "seed-db": "node scripts/seed-database.js",
  "generate-docs": "jsdoc -c jsdoc.json",
  "deploy-staging": "scripts/deploy.sh staging"
}

✅ Version Control Best Practices

package.json and package-lock.json

Always commit both package.json and package-lock.json to version control. The lock file ensures all developers get exactly the same dependency versions.

node_modules

Never commit the node_modules directory to git. Always add it to your .gitignore file:

# .gitignore
node_modules/

Environment-Specific Dependencies

For environment-specific dependencies:

  1. Use .npmrc files for environment-specific configurations
  2. Document required global dependencies in the README.md
  3. Consider using Docker to provide consistent environments

✅ Deployment Considerations

Production Installations

When deploying to production:

npm ci

This command installs exact versions from package-lock.json and is faster for CI environments.

Package Optimization

Before deploying, run:

npm prune --production

This removes development dependencies to reduce deployment size.

Environment Variables

Manage environment-specific variables with:

  • .env files (never commit to git)
  • Environment variables set in your deployment platform
  • The dotenv package to load variables in your application

Important Note

While these NPM standards apply to all DA1 projects, certain specialized projects may have additional requirements. Check the project-specific README.md for any variations from these standards.
